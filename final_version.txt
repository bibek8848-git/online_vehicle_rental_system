3. Project Overview

Secure Drives is a full-stack online vehicle rental platform designed for the Nepalese market. It digitizes the traditional vehicle rental process by allowing customers to register, verify identity, browse approved vehicles, request bookings, and pay digitally through eSewa. It also allows vehicle providers to list vehicles, manage booking requests, and view fleet analytics. Administrators control platform trust by reviewing KYC documents, approving vehicles, and monitoring users, bookings, and payments.

The system addresses problems common in manual rental businesses, such as phone-call based booking, paper records, double-booking, poor transparency, fraud risk, weak identity verification, and cash/manual payment issues. It supports three main roles: Customer/User, Provider, and Administrator.

4. Problem Statement

Traditional vehicle rental services in Nepal often depend on phone calls, cash payment, paper records, and manual identity checks. This creates several issues:
- Customers do not always know which vehicles are available.
- Providers may accidentally double-book the same vehicle.
- Rental records can be lost or incorrectly maintained.
- Identity checking is slow and inconsistent.
- Cash payment creates tracking and transparency problems.
- Trust is low because fake accounts and unverified renters/providers are possible.
- There is no centralized marketplace where customers can compare vehicles from different providers.

Secure Drives solves these problems by providing a centralized web platform with authentication, KYC verification, vehicle approval, date-based booking checks, digital payment tracking, notifications, and dashboards for each role.

17. Google OAuth Configuration

Purpose:
Google OAuth 2.0 allows users to sign in using a Google account.

Package/library:
No NextAuth package is used. The project manually builds OAuth URL and uses fetch() to call Google token and userinfo endpoints.

Files:
- src/app/api/auth/google/route.ts
- src/app/api/auth/google/callback/route.ts
- src/app/SearchParamHandler.tsx
- google-auth-setup.md

Environment variables:
- GOOGLE_CLIENT_ID=your_google_client_id_here
- GOOGLE_CLIENT_SECRET=your_google_client_secret_here
- GOOGLE_REDIRECT_URI=http://localhost:3000/api/auth/google/callback

Flow:
1. User clicks Google login button.
2. Browser goes to /api/auth/google.
3. API redirects to Google OAuth consent URL with scope openid email profile.
4. Google redirects back to /api/auth/google/callback with code.
5. Callback exchanges code for access token.
6. Callback fetches Google user profile from Google userinfo endpoint.
7. System checks if a user with the email already exists.
8. If not found, it creates a new user with name, email, and profile_picture. Password is null.
9. If existing user has no profile picture, it updates profile_picture.
10. System signs JWT with id, email, and role.
11. User is redirected to / with token and user data in query parameters.
12. SearchParamHandler stores token/user in localStorage and cookies.

Role behavior:
- Existing users keep their stored role.
- New Google users are inserted without role, so the database default role USER applies.
- Provider Google registration selection is not connected to the callback for new users. This is a codebase limitation.

- sharpen(): improves character edges.

No explicit denoise/median blur operation found. Mentioned in request but not found in current project files.

Tesseract.js:
- createWorker(['eng', 'nep'], 1, logger/options)
- English and Nepali OCR support is configured.
- Page segmentation mode is set to 6.
- OCR text is parsed with regex.

Extracted fields:
- name
- idNumber
- dob
- address
- expiryDate
- rawText

Fuzzy name matching:
- verifyKYCData() lowercases user and extracted names.
- Splits registered user name into parts longer than 2 characters.
- Counts how many name parts appear in extracted name.
- Approves if matchCount >= Math.min(2, userParts.length).

Automatic approval:
- If fuzzy name match passes, KYC status becomes APPROVED.
- If not, document remains PENDING for manual review with rejection_reason explaining OCR failure/name mismatch.

Admin review:
- Admin can approve/reject KYC through /api/admin/kyc PATCH.
- Updates kyc_documents.status and users.kyc_status.
- Stores rejection reason.
- Creates KYC notification.

KYC gate before booking:
- /api/user/bookings POST checks users.kyc_status must be APPROVED.
- /api/bookings POST also checks KYC status.

Document security:
- KYC documents are stored as BYTEA in PostgreSQL and returned only through protected KYC/admin endpoints.
- Vehicle images are public uploads, but KYC binary document storage is not directly public.
- Additional production hardening such as encryption at rest, file type validation, and size limits was not found in current project files.

Tesseract worker singleton:

Environment variables:

- ESEWA_API_URL=your_esewa_gateway_url_here
- ESEWA_SUCCESS_URL=your_success_url_here
- ESEWA_FAILURE_URL=your_failure_url_here

Sandbox vs production:
- Defaults use eSewa test values and rc-epay URL.
- Production requires live merchant product code, secret key, gateway URL, and callback URLs.

Payment initiation flow:
1. Customer opens payment page for a booking.
2. Frontend fetches booking from /api/user/bookings.
3. Customer clicks Pay with eSewa.
4. Frontend posts bookingId and amount to /api/user/payments/initiate.
5. API requires USER role.
6. API creates transaction_uuid as ESEWA-bookingId-timestamp.
7. Amount is normalized using Number(amount).toString().
8. Signature data is:
   total_amount=amount,transaction_uuid=transactionUuid,product_code=productCode
9. generateEsewaSignature() creates HMAC SHA-256 using ESEWA_SECRET_KEY and base64 digest.
10. A pending payment row is inserted.
11. API returns fields required by eSewa.
12. Frontend dynamically creates a form and posts it to eSewa.

Signed fields:
- total_amount
- transaction_uuid
- product_code

Returned payment fields:
- amount
- tax_amount
- total_amount
- transaction_uuid
- product_code
- product_service_charge
- product_delivery_charge
- success_url
- failure_url
- signed_field_names
- signature
- esewa_url

Success callback:
- /api/payment/success reads base64 data query parameter.
- Decodes JSON.
- If status is COMPLETE:
  - Updates payment status to SUCCESS.
  - Replaces transaction_id with eSewa transaction_code.
  - Updates booking payment_status to PAID.
  - Inserts payment notification for user.

43. How to Explain This Project in Viva

1-minute explanation:
Secure Drives is a full-stack online vehicle rental platform for Nepal. It connects customers with vehicle providers through a secure digital system. Customers can register, complete KYC, browse approved vehicles, book vehicles, and pay through eSewa. Providers can list vehicles, manage booking requests, and view analytics. Admins manage users, KYC, vehicle approvals, bookings, payments, and platform statistics. The system uses Next.js, React, TypeScript, PostgreSQL, JWT, bcrypt, Google OAuth, Tesseract.js, Sharp, and eSewa.

3-minute explanation:
Secure Drives digitizes the manual vehicle rental process. Instead of phone calls, paper records, and cash payment, the platform provides role-based dashboards for customers, providers, and admins. The authentication system uses bcrypt for password hashing and JWT for sessions. KYC documents are uploaded and processed with Sharp and Tesseract.js to extract text, then the system performs fuzzy name matching. If OCR is uncertain, admin manually reviews documents. Providers add vehicles, but vehicles must be approved by admin before customers can see them. Customers can book only after KYC approval. The booking system checks date overlaps to prevent double-booking. Once a booking is approved, the customer can pay through eSewa. Payments, notifications, and analytics are stored in PostgreSQL and shown in dashboards.

Technical explanation:
The project uses a unified Next.js App Router codebase. Frontend pages are under src/app and reusable components are under src/components. Backend APIs are under src/app/api. PostgreSQL is connected using pg Pool in src/lib/db.ts. Authentication helpers are in src/lib/auth.ts and src/app/api/auth/authorization.ts. Middleware in src/middleware.ts protects dashboard and API routes. eSewa signature generation is in src/lib/esewa.ts. OCR logic is in src/lib/ocr.ts.

Database explanation:
The database is PostgreSQL. It has users, kyc_documents, vehicles, bookings, payments, notifications, chat_messages, and reviews tables. Foreign keys connect bookings to users and vehicles, vehicles to providers, payments to bookings/users, and KYC/notifications to users.

eSewa explanation:
The eSewa integration creates a transaction UUID, prepares signed fields total_amount, transaction_uuid, and product_code, signs them with HMAC SHA-256 using the eSewa secret key, stores a pending payment, and sends the user to eSewa using a form. The success callback decodes eSewa response data, updates payment status to SUCCESS, updates booking payment_status to PAID, and creates a payment notification.

Google OAuth explanation:
The app redirects users to Google with openid email profile scope. Google returns a code to /api/auth/google/callback. The backend exchanges the code for an access token, fetches user info, creates or updates the user, signs a JWT, and redirects back with token data. SearchParamHandler stores the token in localStorage and cookies.

KYC explanation:
Users/providers upload document images. The server reads the file buffer, preprocesses it with Sharp, extracts text with Tesseract.js using English and Nepali, parses fields like name and ID number, and compares the extracted name with the registered user name. If it matches, KYC is approved automatically. Otherwise, it stays pending for admin manual review.

Admin explanation:
Admin is seeded from ADMIN_EMAIL and ADMIN_PASSWORD. Admin can view statistics, users, providers, KYC documents, vehicles, bookings, and payments. Admin approves/rejects KYC and vehicles and sends notifications.

Security explanation:
Security is handled through bcrypt, JWT, middleware, role-based API checks, KYC gating, vehicle approval, parameterized queries, and server-side eSewa signature generation. Limitations include missing rate limiting, limited upload validation, and no production-grade eSewa verification endpoint.

44. Common Viva Questions and Answers

Q: Why did you choose this project?
A: Vehicle rental in Nepal is still often manual, phone-based, and cash-based. This project solves a practical local problem using a secure digital platform.

Q: What problem does it solve?
A: It reduces manual booking, double-booking, poor record keeping, fraud risk, weak identity verification, and payment tracking problems.

Q: Why PostgreSQL?
A: The system has relational data: users, providers, vehicles, bookings, payments, KYC, and notifications. PostgreSQL supports foreign keys, transactions, constraints, and reliable structured queries.

Q: Why Next.js?
A: Next.js supports both frontend pages and backend API routes in one project. It is suitable for a full-stack web app with dashboards and APIs.

Q: Why TypeScript?
A: TypeScript improves code safety by catching type errors during development and making data structures easier to understand.

Q: Why JWT?
A: JWT allows stateless authentication. The token stores user id, email, and role, so APIs and middleware can identify and authorize users.

Q: Why bcrypt?
A: bcrypt securely hashes passwords with salt. The project uses salt rounds 10, which is a common balance between security and performance for development-level applications.

Q: How does Google OAuth work?
A: The user is redirected to Google, Google returns an authorization code, the backend exchanges it for an access token, fetches profile data, creates or updates the user, creates a JWT, and redirects the user back to the app.

Q: How does eSewa integration work?
A: The backend creates payment data and signs selected fields using HMAC SHA-256 with the eSewa secret key. The frontend submits a form to eSewa. On success, the callback updates payment and booking status.

Q: How do you prevent double booking?
A: Before creating a booking, the API checks existing bookings for the same vehicle where status is PENDING or APPROVED and where date ranges overlap. If a conflict exists, it returns 409.

Q: How does KYC verification work?
A: A document image is uploaded, preprocessed with Sharp, read with Tesseract OCR, parsed for fields, and compared with the registered name. If the match is strong, it is auto-approved; otherwise admin reviews it.

Q: Is OCR an AI feature?
A: It is OCR-based automation using Tesseract.js. It extracts text from images. The project does not train a machine learning model or use an AI language model in current code.

Q: What are the main security features?
A: bcrypt password hashing, JWT authentication, middleware protection, role-based access control, KYC verification, vehicle approval, double-booking prevention, parameterized queries, and server-side eSewa signature generation.

Q: What are the limitations?
A: OCR depends on image quality, chatbot is rule-based, real-time notifications use polling, production eSewa verification needs improvement, map search is missing, report export is missing, and upload validation should be stronger.

Q: What would you improve in the future?
A: Add WebSocket notifications, mobile app, map search, Khalti/ConnectIPS, stronger payment verification, PDF/Excel reports, review UI, refund workflow, better OCR templates, and advanced filters.

45. Conclusion

Secure Drives is a complete full-stack online vehicle rental system for Nepal. It includes multi-role authentication, KYC verification, provider vehicle management, booking workflow, eSewa payment processing, admin monitoring, notifications, provider analytics, chatbot support, and secure API/database integration. The system is practical for Nepal because it supports digital booking, eSewa payments, identity verification, and centralized vehicle rental management. The current codebase is strong for a final year project, while future improvements can make it more production-ready.
